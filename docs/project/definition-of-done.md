# وثيقة معايير الاكتمال (Definition of Done - DoD)
## منصة SaaS الذكية لإنشاء دراسات الجدوى بالذكاء الاصطناعي (Feasibility Suite)

---

تحدد هذه الوثيقة الشروط والمعايير الصارمة التي يجب استيفاؤها قبل اعتبار أي ميزة، أو كود برمجي، أو مرحلة من مراحل المشروع "مكتملة" بنسبة 100% وجاهزة للنقل للمرحلة التالية.

---

### 1. معايير الاكتمال العامة للأكواد البرمجية (General Code Quality DoD)

قبل دمج أي كود برمجي (Pull Request - PR) في الفرع الرئيسي لـ Git (`main` / `master`)، يجب استيفاء الآتي:
- [ ] **خلو الكود من التحذيرات والأخطاء:** لا توجد أخطاء في الـ Compiler أو الـ Build.
- [ ] **مراجعة الأكواد (Code Review):** تمت مراجعة الكود والموافقة عليه من قبل مهندس برمجيات آخر على الأقل.
- [ ] **التوافق مع أدوات الفحص (Linting & Formatting):** يمر الكود بنجاح عبر اختبارات ESLint و Prettier لتوحيد نمط التنسيق.
- [ ] **التعامل مع متغيرات البيئة:** لا توجد أي قيم سرية (مثل مفاتيح الـ API أو كلمات المرور) مكتوبة مباشرة في الكود (No Hardcoded Secrets)، ويتم استدعاء جميع الإعدادات من ملفات `.env`.
- [ ] **حفظ تعليقات الكود والتوثيق:** تم توثيق الدوال المعقدة وكتابة تعليقات توضيحية وتحديث وثيقة الـ API في حال تعديل أي مسارات.

---

### 2. معايير اكتمال كل مرحلة من مراحل المشروع (Phase-Specific DoD)

#### **المرحلة الأولى: التحليل والتخطيط (P1 - Analysis)**
- [ ] تمت صياغة وثائق المتطلبات الأساسية (العمل والمستخدمين والمعايير).
- [ ] تمت الموافقة على التصميم المعماري واختيار التقنيات من الفريق التقني والعميل.
- [ ] تم إنشاء مستودعات GitHub وإضافة أعضاء الفريق ومنح الصلاحيات اللازمة.

#### **المرحلة الثانية: تصميم واجهات المستخدم (P2 - UI/UX)**
- [ ] توفير واجهات Figma كاملة لجميع الشاشات في الوضعين (الويب ومتجاوبة مع الجوال).
- [ ] إتمام تصميم رحلة المستخدم بالكامل من صفحة الهبوط لمعالج البيانات حتى صفحة الدفع وتنزيل الـ PDF.
- [ ] اعتماد نظام الألوان، والخطوط والمكونات الجاهزة (UI Kit/Design System) للتأكد من تناسق المظهر.

#### **المرحلة الثالثة: تطوير الواجهة الأمامية (P3 - Frontend)**
- [ ] تحويل تصاميم Figma بدقة عالية إلى واجهات React بالاعتماد على Tailwind CSS.
- [ ] التحقق من صحة المدخلات في جميع النماذج وصلاحية الحقول الرقمية قبل الإرسال.
- [ ] التأكد من توافقية العرض وسرعة التصفح وتجاوب الواجهة مع مختلف أحجام الشاشات ومتصفحات الويب الرئيسية (Chrome, Safari, Firefox, Edge).

#### **المرحلة الرابعة: تطوير الواجهة الخلفية (P4 - Backend)**
- [ ] اكتمال بناء جميع الـ Endpoints وتجربتها بنجاح عبر برمجيات الفحص (مثل Postman).
- [ ] بناء وتدقيق منطق التحليلات والعمليات الحسابية المالية واختبار دقتها الرياضية.
- [ ] كتابة برمجيات الحماية مثل التحقق من صحة الـ Tokens (JWT middleware) والتحقق من صحة المدخلات (Zod validator middleware).

#### **المرحلة الخامسة: دمج الذكاء الاصطناعي وبوابة الدفع (P5 - Integrations)**
- [ ] اختبار اتصال خادم Express مع OpenAI وتوليد التحليلات وحفظ النتائج في قاعدة البيانات كـ `JSONB`.
- [ ] تكامل خادم التطبيق مع Stripe checkout لإنشاء الجلسات ومعالجة الدفع واستقبال الـ Webhooks لتحديث خطط المستخدمين تلقائياً.
- [ ] إتمام برمجة مولد الـ PDF وضمان خروج الملف بتنسيق عربي سليم (RTL) متوافق مع شاشات الطباعة.

#### **المرحلة السادسة: اختبار الجودة والأمان (P6 - QA & Testing)**
- [ ] تنفيذ اختبارات التكامل الشاملة (End-to-End Testing) لرحلة مستخدم يسجل ويشترك وينشئ دراسة ويقوم بتحميلها.
- [ ] خلو النظام من أي أخطاء حرجة (Critical / Blocker Bugs).
- [ ] التحقق من أمان الخادم وحمايته ضد الهجمات الشائعة (مثل SQL Injection, XSS) عبر استخدام Helmet وتدقيق حقول البيانات.

#### **المرحلة السابعة: الإطلاق والنشر (P7 - Deployment & Launch)**
- [ ] نجاح عملية النشر (Production Deployment) على Vercel و Render بدون توقف أو مشاكل.
- [ ] ربط نطاق الويب الرسمي (Domain Name) وإعداد شهادة الـ SSL (HTTPS).
- [ ] تفعيل نظام النسخ الاحتياطي التلقائي اليومي لقاعدة البيانات وتجربة استعادتها بنجاح.
- [ ] تفعيل أدوات تتبع الأخطاء ومراقبة الخادم (مثل Sentry أو LogRocket) للتدخل الفوري عند حدوث مشاكل في الإنتاج.

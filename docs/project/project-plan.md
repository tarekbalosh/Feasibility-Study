# وثيقة خطة المشروع (Project Plan Document)
## منصة SaaS الذكية لإنشاء دراسات الجدوى بالذكاء الاصطناعي (Feasibility Suite)

---

### 1. الملخص التنفيذي (Executive Summary)

تهدف هذه الخطة إلى توجيه عملية تطوير وإطلاق منصة **Feasibility Suite** خلال فترة زمنية محددة بـ **14 أسبوعاً**. يتمحور المشروع حول تقديم منصة برمجية كخدمة (SaaS) تعتمد على الذكاء الاصطناعي لتوليد دراسات الجدوى والتقارير المالية والتشغيلية لرواد الأعمال والمشاريع الصغيرة والمتوسطة. تم تقسيم المشروع إلى **7 مراحل رئيسية** متداخلة ومتكاملة لضمان الكفاءة القصوى واستغلال الموارد بشكل مثالي، والانتقال بسلاسة من مرحلة التخطيط والتحليل إلى الإطلاق التجاري الفعلي.

---

### 2. نطاق العمل (Project Scope)

#### ما هو داخل النطاق (In-Scope):
* تطوير بوابة الدخول والمصادقة الآمنة (JWT) والتسجيل عبر حسابات Google.
* تطوير معالج إدخال البيانات المالي والتشغيلي المكون من 5 خطوات تفاعلية.
* بناء محرك التحليل المالي والرياضي لحساب المؤشرات (IRR, NPV, Break-even, Payback Period).
* دمج وتدريب محرك ذكاء اصطناعي (OpenAI API) لتوليد نصوص تحليلات السوق والمنافسين وتحليل SWOT.
* توفير ميزة تصدير التقارير النهائية بصيغة PDF قابلة للطباعة والتداول.
* دمج بوابة الدفع الإلكتروني (Stripe) لإدارة الاشتراكات الشهرية وتوليد الفواتير.
* توفير نظام مشاركة آمن عبر روابط فريدة للقراءة فقط (Read-only URLs).

#### ما هو خارج النطاق (Out-of-Scope - للنسخ المستقبلية):
* تطبيق للهواتف الذكية (iOS / Android) - سيتم التركيز على واجهة ويب متجاوبة (Responsive Web App).
* تقديم استشارات مباشرة وحية مع خبراء بشريين داخل المنصة.
* ربط المنصة ببيانات السوق والأسعار اللحظية الخارجية عبر الـ APIs الجغرافية.
* إمكانية قيام المستخدمين بتصميم قوالب دراسات الجدوى بأنفسهم من الصفر.

### 4. التبعيات بين المهام (Dependencies)

لضمان عدم توقف العمل، تم تحديد التبعيات الحرجة كالتالي:
* **UI/UX Design (P2) ➔ Development (P3/P4):** لا يمكن البدء في كتابة كود الواجهات الأمامية إلا بعد اعتماد تصاميم Figma بنسبة 100%.
* **Prisma Schema & Database Setup (P4) ➔ Backend Routes (P4):** يجب إتمام تهيئة جداول قاعدة البيانات والربط معها قبل برمجة مسارات الحسابات المالية أو حفظ المشاريع.
* **Backend Core API (P4) ➔ Frontend API Integration (P3):** تعتمد الواجهة الأمامية على اكتمال نهايات الـ API بالخلفية لربط البيانات الحية وتجربة المستخدم الفعلية.
* **AI Prompts & Integration (P5) ➔ QA (P6):** يجب استقرار نموذج الـ Prompt وإعداد OpenAI Structured Outputs قبل البدء في اختبارات القبول الشاملة للتأكد من جودة النصوص.
* **Stripe Webhook Verification (P5) ➔ Phase 7 Launch:** يجب التحقق من استجابة Webhooks وموثوقيتها في بيئة الإنتاج قبل فتح التسجيل المدفوع للمستخدمين.

### 6. المخاطر المحتملة وخطط التخفيف (Risks & Mitigations)

1. **المخاطر: تأخر استجابة OpenAI أو تغير هيكل الـ JSON المرتجع.**
   * *خطة التخفيف:* استخدام ميزة `Structured Outputs` المتاحة في API الخاص بـ OpenAI لفرض هيكل محدد (Schema) للاستجابة، مع كتابة برمجيات وسيطة لإعادة المحاولة (Retry Mechanism) والتعامل الذكي مع انقطاع الخدمة عبر تقديم نصوص بديلة مؤقتة للمستخدم.
2. **المخاطر: بطء حساب المؤشرات المالية للمشاريع المعقدة وتأخر استجابة الخادم.**
   * *خطة التخفيف:* تنفيذ جميع العمليات الحسابية المالية محلياً داخل خادم Node.js باستخدام مكتبات رياضية خفيفة وسريعة، وعدم إسناد أي عمليات محاسبية للذكاء الاصطناعي لضمان السرعة والدقة التامة للأرقام.
3. **المخاطر: تسرب بيانات المشاريع الحساسة وأفكار المستخدمين.**
   * *خطة التخفيف:* تشفير حقول البيانات المدخلة في قاعدة البيانات (AES-256)، وفرض قيود صارمة على الـ CORS لمنع الاستدعاءات الخارجية للـ APIs، وعمل تدقيق دوري لصلاحيات الـ JWT.

### 7. الافتراضات والقيود (Assumptions & Constraints)

* **الافتراضات:**
  * توفر حسابات مطورين نشطة لخدمات Stripe و OpenAI برصيد شحن كافٍ قبل الأسبوع 9.
  * توافق بيئات التشغيل والاستضافة (Vercel & Render) مع متطلبات استقرار التطبيق.
  * استقرار أسعار خدمات الذكاء الاصطناعي OpenAI ضمن الميزانية المرصودة للمشروع.
* **القيود:**
  * حدود استهلاك الـ API (Rate Limits) المفروضة من OpenAI قد تقيد عدد الدراسات المولدة لكل دقيقة في البداية.
  * التقرير المالي يعتمد على مدخلات المستخدم؛ المنصة لا تضمن نجاح فكرة المشروع قانونياً بل تقدم تحليلاً مبنياً على الافتراضات المدخلة.
